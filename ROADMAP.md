## Bug 1 Resolution — Option 3 Durable Write Queue ✅ (shipped 2026-05-13)

*Bug 1 was originally tracked as LT2 in Phase 0.5. The root cause turned out to be silent write failures, not a hole-nav regression. Option 3 closes it durably with a localStorage-backed write queue.*

**Design doc:** [`docs/option-3-write-queue-design.md`](./docs/option-3-write-queue-design.md). Captures all 14 locked decisions (D1–D14): storage layer, namespace, queue item shape, collapsing rules, DB conflict resolution, retry / backoff policy, failure classification, UI flow (invisible during play; reconciliation on End Round; on-open stale-failure prompt), drain triggers, multi-tab handling, pre-drain on pagehide, quota handling, Sentry instrumentation.

**What shipped:**

| Phase | Commit | What it does |
| --- | --- | --- |
| A | `08ecdce` | `setScore` switched from SELECT-then-INSERT/UPDATE to a single `.upsert(..., { onConflict: "round_player_id,hole_number" })`. One round-trip, idempotent under retry. No new migration — Phase 2's audit had missed that `scores_round_player_id_hole_number_key` already existed (predates `supabase/migrations/`; `list_tables` doesn't surface unique constraints). |
| B | `116801f` | New `src/lib/writeQueue/` module: `WriteQueue` class + `localStorage`-backed storage + backoff helper. Scaffolding only, not imported by anything yet. Full unit test coverage. |
| C | `cc0d773` | Wires the queue into the scorecard. `setScore` enqueues instead of awaiting. `load()` drains the queue *before* rehydrating, then overlays any still-pending or in-flight items onto `scoreMap`. Production `instance.ts` singleton wraps `@sentry/nextjs` as the reporter. |
| D | `76e1a8b` | End-Round reconciliation flow. Tap → spinner → hail-mary drain → 30s timeout with "Skip and finish" at 15s → first-attempt dialog (`Retry sync` / `Skip and finish`) → second-attempt dialog (`Try Again` / `Copy details` / `Finish anyway`). New `markAsTerminal(ids, reason)` API. |
| E | `668da1e` | Homepage on-mount stale-failure prompt. `Retry` / `View details` / `Forget`. `sessionStorage`-scoped dismissal. Multi-round-grouped clipboard format. |

**Failure mechanisms closed:**

| Mechanism | Closed by |
| --- | --- |
| Write failures mid-flight | Phase B retry loop with exponential backoff (1s → 120s, then 120s forever during round) |
| Tab eviction during write | Phase C localStorage persistence + Phase C `load()` overlay |
| Offline-on-remount | Phase C overlay reapplies pending queue items onto `scoreMap` before render |
| End-of-round recovery | Phase D dialog surfaces stuck items before the user leaves the scorecard |
| Multi-session recovery | Phase E prompt surfaces terminal items left over from prior session |

**Code locations** (one-paragraph orientation for future extenders):

- `src/lib/writeQueue/` — the queue itself. `WriteQueue.ts` is the class. `instance.ts` is the production singleton that wraps Supabase upsert + `@sentry/nextjs`. `storage.ts` is the localStorage adapter with in-memory fallback and quota-eviction. `backoff.ts` exports the D7 schedule constants. `types.ts` is the public type surface. `index.ts` re-exports everything.
- `src/components/scorecard/` — the dialogs. `ReconciliationDialog.tsx` is Phase D's End-Round dialog (both variants). `FinishingSpinner.tsx` is the spinner. `StaleFailureDialog.tsx` is Phase E's homepage prompt. `stuckItemsClipboard.ts` has both formatters (single-round for Phase D, multi-round-grouped for Phase E).
- `src/app/round/[id]/scorecard/page.tsx` — Phase C wiring (setScore enqueues, load drains+overlays) + Phase D handlers (startEndRoundFlow, runHailMaryWithTimeout, handleRetrySync, handleSkipAndFinalize, handleCopyDetails, finalizeRound).
- `src/app/page.tsx` — Phase E homepage mount-check + handlers.
- `tests/lib/writeQueue/` — unit tests for the queue module.
- `tests/components/scorecard-bug-repro.test.tsx` — Phase 3 + Phase C bug-repro sequences A–G.
- `tests/components/end-round-flow.test.tsx` — Phase D integration tests.
- `tests/components/stale-failure-homepage.test.tsx` — Phase E integration tests.

**How to extend:**

- **New queue write kinds:** the `QueueItem.kind` discriminator is set up to support more than `"score_upsert"`. Add a new kind, extend `WriterFn` to dispatch on it, and Phase A's upsert call moves out of `instance.ts`'s closure into a per-kind dispatcher.
- **New drain triggers:** add an event listener inside `WriteQueue.start()` and call `void this.drain()`. The existing triggers (online / visibilitychange / pageshow / pagehide / 30s backstop / enqueue) cover the practical cases.
- **New Sentry events:** add a `captureMessage` call inside `WriteQueue` with a descriptive event name and `reason` discriminator. Phase E's `"user_forget_stale"` is the model.
- **Multi-tab live sync** (deferred Phase F per the design doc): would require Supabase Realtime subscriptions. Out of scope for Bug 1 closure; revisit only if admin laptop + player phone editing the same scorecard concurrently becomes a real workflow.

**Sentry instrumentation surface** (per D14): `writeQueue.terminal_failure` (with `reason` discriminator: `classified_terminal`, `stuck_too_long`, `end_round_timeout`, `end_round_retry_timeout`, `stale_failure_retry_timeout`), `writeQueue.evicted` (storage quota), `writeQueue.writer_threw` / `writeQueue.storage_save_failed` / `writeQueue.listener_threw` (drain-loop crashes), `writeQueue.size_threshold_exceeded` (queue ≥100 items), and `writeQueue.forget` (with `reason: "user_forget_stale"` for Phase E forget taps; `reason: "forget"` for any other future caller). Routine ops (every enqueue / drain / retry) are intentionally NOT logged to avoid drowning the signal.

**Test surface added:** 22 tests in `tests/components/StaleFailureDialog.test.tsx` + `tests/components/stale-failure-homepage.test.tsx` + multi-round formatter (Phase E); 20 in `tests/components/end-round-flow.test.tsx` + `ReconciliationDialog.test.tsx` + clipboard formatter (Phase D); 27 in `tests/lib/writeQueue/{backoff,storage,WriteQueue}.test.ts` (Phase B). Plus the Phase 3 component tests carrying Sequences A–G in `scorecard-bug-repro.test.tsx`. **Total master: 240/240 passing.**

### Terminology

*Locked 2026-05-08, ahead of first live golf-course test. Supersedes the prior "Strokes terminology replaces CH/HC/HCP" decision from Phase A — that rename was a half-step; the bare "Strokes" label was ambiguous (it was being used for course_handicap on the scorecard but for handicap_index in the admin Players tab). Two distinct quantities, one label. Now disambiguated.*

- **Handicap Index (HI):** Player's portable rating, stored in the `handicap_index` column on `players`. Same value across all tees. WHS calculation; player attribute not round attribute.
- **Course Handicap (CH):** Slope-adjusted strokes received on a specific tee, stored in the `course_handicap` column of `round_players`. Round attribute, computed via `computeCourseHandicap` from the player's per-round HI (`round_players.handicap_index_snapshot` as of H.2.5) plus the round's selected tee slope/rating/par. Recomputed on scorecard load (LT1 fix, 2026-05-09) so admin edits to HI on active rounds flow through to the displayed CH without manual intervention; downstream consumers (summary, leaderboard, season) read the value the scorecard wrote back. **Self-heal reads from `round_players.handicap_index_snapshot` and only fires on rounds where `is_complete = false`** (H.2.5.4, 2026-05-21). (The earlier May 8 entry referenced a `player_course_handicaps` table — that table does not exist; `round_players.course_handicap` is the actual storage.)
- **Display rule:** UI never uses bare "Strokes" or "Handicap" alone. Always qualified as "Handicap Index" (or short "HI" where space is tight) or "Course Handicap" (or short "CH"). Database column names (`handicap_index`, `course_handicap`) are already correct and stay as-is.

- **Format authority:** Admin only. Players cannot pick or change.
- **Scorecard pre-format:** Locked until format chosen. Display "Waiting for format" message.
- **Stat display drop:** Gross removed from scorecard top pill. Kept everywhere else.
- **Stroke-allocation dots:** On the per-hole entry surface, handicap strokes received on the current hole are shown as small navy dots above the +/− buttons. No numeric overlay alongside.

### Player-driven team formation (locked 2026-05-20)

Decisions from Phase A.2. Don't relitigate without explicit conversation.

- **Picker scope:** The `PlayerPickerSheet` shows the full unassigned-player pool, not just played-with matches. Played-with history informs the smart-join ranking but all players are selectable.
- **User must tap themselves AND teammates:** A player joining a team taps their own name first (self-selection), then optionally taps teammates to add them. There is no "add player X to team Y" flow for players — only admins drag-and-drop.
- **Smart-join branches:** Three paths: (a) player taps self, no team → `smartJoin` recommends a team or "form new" based on played-with matrix + HI spread; (b) player taps teammate card while in a team → adds that player to the tapper's team; (c) admin can override freely via existing Edit Teams drag-and-drop.
- **Removed player → `team_number = 0`:** Removing a player from a team sets `round_players.team_number = 0` (unassigned), not deletion. The `round_players` row stays; the player re-appears in the unassigned pool.
- **Manage Team hides on first score:** The "Manage Team" button on the scorecard is hidden once any score is entered for the team (`holesPlayed > 0` on any team member). Prevents mid-round roster changes that would break scoring history.
- **Race-condition write queue:** All `round_players` writes from the picker and Manage Team sheet use the per-player write queue pattern (see Locked patterns in CLAUDE.md). Rapid tap patterns (pick player → assign to team) serialize per-player to prevent silent UPDATE-matches-0-rows failures.
- **Bottom-sheet / modal pattern:** `PlayerPickerSheet` and `ManageTeamSheet` are bottom sheets on mobile (<768px) and centered modals on desktop. Consistent with the `FormatPicker` pattern established in Phase B.
- **Device identity deferred (I15):** The picker intentionally ships without per-device player identity. Players select themselves manually each time. Device-local `localStorage` identity (I15) is a deferred improvement; don't build it until post-launch friction proves it's worth the complexity.

### Format set (locked 2026-05-10)

Five formats: **2-Ball, 3-Ball, Best Ball, Stableford Standard, GOBS Stableford**. GOBS House dropped from codebase, format enum, UI, tests, snapshots, and roadmap. `stableford_modified` renamed to `gobs_stableford` (same engine machinery, new defaults + UI). ### Stableford point values (locked 2026-05-10)

Two distinct Stableford formats with separate point tables. *Standard's table is locked at the engine constant and intentionally has no admin UI. GOBS Stableford's defaults are editable per round via FormatPicker (A1.5) — overrides land in `format_config.point_values`.*

**Stableford Standard** (locked, NOT editable):
- Albatross: +8
- Eagle: +5
- Birdie: +3
- Par: +2
- Bogey: +1
- Double Bogey or worse: 0

**GOBS Stableford** (defaults, editable per-round):
- Albatross: +8
- Eagle: +5
- Birdie: +3
- Par: +2
- Bogey: 0
- Double Bogey or worse: −1

Implications: GOBS Stableford can produce negative team totals when many double-bogey-or-worse holes accumulate; leaderboard rank + `formatTeamTotal` handle negative `pts` via Unicode minus. `getStablefordPoints` collapses any net result of +2 or worse to the doubleBogeyOrWorse bucket (engine confirmed via `delta >= 2` final return).

### Best Ball format (locked 2026-05-09, override behavior clarified 2026-05-10)

- **Selection rule:** strict best-1 — exactly one player's score counts per hole, regardless of team size.
- **Scoring basis:** net only. Format's purpose is handicap equalization; gross best ball undermines the equalizer in a mixed-handicap league. Net/gross toggle disabled in the picker for this format and labeled "Best Ball is always net." FormatPicker also force-flips local state back to "net" any time Best Ball is selected so a stale "gross" choice can't slip through.
- **Override-holes:** apply normally. Best Ball is a best-N (N=1) family format, so an override turns the hole into "all scores count" exactly as it does for 2-Ball / 3-Ball.
- **Engine:** `computeBestNHole` with N=1 via `defaultBestN("best_ball") === 1`.

### Format selection entry point (locked 2026-05-10, supersedes 2026-05-09)

Admin opens the Round Setup tab and sees two top-level buttons: "Today's Format" and "Edit Teams". Either button can be tapped first; both auto-create a round shell for today if none exists. Format and team-building are fully independent — admin can set format with no teams built, or build teams with no format set. Format applies live to every scorecard for the day, regardless of creation order. Players starting scorecards from the homepage trigger round-shell auto-creation too; their scorecard locks with ScorecardLockNotice until admin sets format. No format UI of any kind appears on player surfaces.

### Non-paying player handling (locked 2026-05-09)

Players who opt out of betting are not put on a scorecard. Their team plays as a blind-draw team. No app-side toggle needed for "exclude from betting but keep scoring."

### Handicap Index snapshot (locked 2026-05-20)

- **Snapshot at insert.** Every round_players row stores the handicap_index that was in effect for the player on that round, in a dedicated round_players.handicap_index_snapshot column. All course-handicap math reads from the snapshot, not from the current players.handicap_index.
- **Finalized rounds are immutable.** Once a round has is_complete = true, its stored CH and snapshot HI never change. The LT1 self-heal is gated on is_complete = false.
- **Active rounds auto-cascade.** When admin edits a player's HI in the Players tab, the snapshot is automatically updated for every round_players row where the player is currently in an in-progress round (is_complete = false). This handles both the correction case ("I forgot to update before tee-off") and the mid-round-update case. Finalized rounds untouched.
- **No mid-round affordance needed.** Admin doesn't need a separate "apply to current round" button. The HI edit IS the application.
- **Historical import uses the same column.** Phase H.5 importer writes the per-round HI Dad provides directly to handicap_index_snapshot. No special path; same column, same math.
- **Backfill: copy current HI to snapshot.** Existing round_players rows pre-H.2.5 had their CH computed from the player's then-current HI, which happens to equal current HI for all existing prod rounds (no HIs have changed yet). Backfill is therefore lossless.

### Played With v2 (locked 2026-05-24)

Decisions for the Phase E v2 season-scope filter (E5) and admin redesign (E6). Don't relitigate without explicit conversation. v1 (E1, `d506460`) is built against these same decisions; v2 inherits.

- **"Played with" = same team, same round.** Not same day. The round-and-team pair is the unit of partnership. Two players on different teams the same day are not "played with."
- **Bucket thresholds: 6+ / 3–5 / 1–2 / 0.** Same as v1. Don't recalibrate buckets per season — they're stable across All-time and This-season filters.
- **Deactivated players:** Shown in the relevant bucket if they share history with the focal player (historical fact). Hidden from the Never-played bucket (deactivated players aren't actionable — admin can't currently pair them).
- **Display name disambiguation:** Handle at the DB layer (full_name / display_name choices in the players row), not at render time. No render-time suffixing of duplicate names.
- **All-time empty bucket label:** *"Not yet"* in italic muted text. Same as v1.
- **Pills carry `player_id`, navigate to that player's page.** `<Link href={`/player/${id}`}>` is the locked pattern. Navigation must be ID-keyed to survive future display-name edits.
